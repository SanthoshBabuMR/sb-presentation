Fluid Layout

When it comes to desktop layout, we rely on constraints; especially 'screen resolution', which needs to be set to a specific width for our layout.

With variety of desktop screen's out there, we tend to focus on the majority of the population with a common screen resolution, which till now was 1024 x 768.
ref: http://gs.statcounter.com/#resolution-ww-monthly-201012-201202

However, current statistics reveal that 1366 x 768 has overtaken 1024 x 768 as the most popular screen resolution and this too will change in the near future.
ref: http://gs.statcounter.com/#resolution-ww-monthly-201103-201203

Till now we have focused on designing websites using a fixed layout width of 960px. This implied that, desktop users with lower resolution would have to deal with horizontal scrollbar and users with higher resolution would be presented with a relatively smaller looking website.

Designing for a fixed width screen resolution wouldn't be an ideal solution, as we could see large widescreen displays being introduced at one end of the spectrum and we have the narrow screen devices (mobile devices) at the other extreme end. In between the two extremes, we have smart phones and tablets.

The way forward would be to design layout(s) in proportion to the available screen size. This is best accomplished by using percentage (%) as a unit of measurement for width. By designing the layouts in percentages, we can have our containers/layout stretch in proportion to the available screens size. Every aspect of our layout - the rows and columns, and the modules draped over them -can be ex-pressed as proportions of their containing element, rather than using unchanging, inflexible pixels.

This involves a little bit of the following math,

target / context = result

  • target = parent container;
  • context = child container;

Fluid layout requires fluid width, fluid margins and fluid padding; so we set width, padding and margin in percentages

If we consider the above PSD, the mockup code is as follows,

HTML Markup <div id="page-wrapper"> <div id="page-header"> <header> sitename </header> </div> <div id="page-content" class="clrfix"> <section class="blogpost"> blogpost </section> <aside class="secondary"> aside content </aside> </div> <div id="page-footer"> <footer> copyright info </footer> </div> </div>

CSS #page-wrapper{ width: 100%; /* 1020px/1020px */ } #page-header, #page-footer{ width: 100%; /* 1020px/1020px */ } header, footer{ width: 94%; /* 960px/1020px */ } #page-content{ width: 94%; /* 960px/1020px */ } .blogpost{ width: 62.5%; /* 600px/960px */ } .secondary{ width: 37.5%; /* 360px/960px */ }

Setting Width:

target / context = result

When setting flexible width, the target is the current element, while context refers to its parent container.

So, calculating width for header,

header [width: 960px;] is target
#page-header [width: 1020px;] becomes context,

Hence, width = 960px / 1020px = 94.11%;

Similarly considering, .blogpost container,

.blogpost [width: 330px;] is target
#page-content [width: 960px;] becomes context

Hence, width = 330px / 960px = 37.5%;

Note that context keeps changing as it represents the parent of the current target element.

Setting Margins & Paddings:

We could use a combination of em's and percentage's to set margins and paddings.

em unit is based on font size, while percentage is based on width of the parent element.

If percentages are used, the following applies,

1. When setting flexible margins on an element, the context is the width of the element's parent container.
2. When setting flexible padding on an element, the context is the width of the element itself.

For a parent container of width 960px, containing a child element, whose margin needs to be set at 15px;

With reference to percentage(s)
- margin for the child element would be set to 15px / 960px = 1.5625%; /* context is the width of the parent element */

In case of em,
- margin for the child = 15px / 13px = 1.15em /* context is font-size of parent element */

If the child element had a width of 400px,

With reference to percentage(s)
- padding for the child element would be set to 10px / 400px = 2.5%; /* context is the width of the element itself */

In case of em,
- padding for the child = 10px / 13px = 0.769em /* context is font-size of parent element */

The difference between percentage and em is that, as we resize the browser or zoom-in/zoom-out, values set based on percentage, will reflow accordingly, while values set based on em will appear constant.

Fluid Media

Fluid media is about making flexible images and flexible videos that could respond to its environment.

Looking back at the web design community, we already had fluid layouts. The fluid layouts were good at reflowing text according to the available width. But it had one problem, only the layout and text were flexible; when it came to images, it would often break the layout, when the viewport was either smaller or larger.

Responsive Web Design is the next-step to fluid layouts; We now have the power to resize images and videos, making the most of the web page components to be truly responsive.

The key to making images and videos flexible is the CSS max-width property.

CSS Max-Width img{ max-width: 100%; }

With the above CSS in place, the img element will render at whatever size it wants, as long as it's narrower than its containing element. But if it happens to be wider than its container, then the max-width: 100% directive forces the image's width to match the width of its container.

Modern browsers have evolved to the point where they resize the images proportionally, as the flexible container resizes itself, shrinking or enlarging the image, with the image's aspect ratio remains intact

When it comes to browser support for max-width, Internet Explorer 6 and below don't support the max-width property, while IE7 and higher, do have support for this property.

So, a simple approach would be to set width: 100% to IE6 Specific styles, while the rest of the browsers get the max-width property.

IE - Setting Img Width img{ width: 100%; }

The downside to this approach is that, it forces the images to always match the width of their con-taining elements. As a result, smaller images would be forced to scale to 100% of the parent container.

To overcome this, we could have a CSS class applied to target only the required images as,

IE - Setting Img Width Using Class img.large{ width: 100%; }

Windows & Image Resizing:

Windows doesn't scale images that well. In fact, when resized via CSS, images quickly develop artifacts on Windows, dramatically impacting their quality. However, this effect is limited to IE6,7 and Firefox 2.

AlphaImageLoader, one of Microsoft's proprietary CSS filters dramatically improves the rendering quality of images in IE.

By including the imgSizer.js script based on AlphaImageLoader, we can address the rendering quality issue in IE6 & 7.

ref: http://unstoppablerobotninja.com/entry/fluid-images

However, there is performance tradeoff when using AlphaImageLoader and it could negatively impact the performance of the website. So, it would be necessary to evaluate whether or not the improved rendering is worth the performance tradeoff.
ref: http://www.yuiblog.com/blog/2008/12/08/imageopt-5/

Images rendered using img tag can be made flexible by using CSS max-width; however, images rendered as background needs be addressed in a different approach.

Faux Column:

Faux Column technique employs background-image to create the illusion of equal height columns; however, it is restricted to fixed width layout.

By using the little math, target context = result, we can achieve flexible faux column layout.

First, we find out the transition point in our background image, the exact pixel at which our background image transitions from one color to another.

Say, we have a 960px [context] layout, and exactly at 576px [target], the background-image transition to a different color. Now, we apply the math,
576px / 960px = 60%

We can now move to Photoshop, create a blank canvas of say 3000 pixels wide and 20 pixels in height. We require a large canvas, to accommodate large screens.

we convert 60% to pixels, which is 3000 x (60/100) = 1800px; and have the corresponding background-color applied, at this transistion point.

The result of using this approach is that, we end up with a faux column background image that resizes accordingly.

Background-Images:

When it comes to background-images, we can identify which part of the background-image stresses the importance, and ensure that portion of the background-image is visible at all times, using background-position property. CSS3's background-size property can also be used to achieve this.

Media Queries

Using fluid layouts, fluid media, we have ensured the layout doesn't break when the page is scaled-up or scaled-down.

However, there is a threshold limit at which scaling too high to scaling too low, would break up the design. In other words, as we move beyond the resolution for which it was originally designed, the layout becomes a liability to our content. Its proportions constrict the content at smaller resolutions, and isolate it in a sea of whitespace at higher ones.

This isn't a problem unique to flexible layouts. No design, fixed or fluid, scales well beyond the context for which it was originally designed.

Media Queries introduced by W3C, helps in creating design that is not only flexible, but that can adapt to changes in screen resolution and viewport dimensions.

We can use media queries to optimize the display of our content to best meet the needs of the device, creating alternate layouts tailored to different resolution ranges. By conditionally loading style rules that target these ranges, media queries allow us to create pages that are more sensitive to the needs of the devices that render them.

Media Query has two components:

  1. - Each query begins with a media type (screen), drawn from the CSS2.1 specification's list of approved media types (http://www.w3.org/TR/CSS21/media.html#media-types).
  2. - Immediately after comes the query itself, wrapped in parentheses: (min-width: 1024px).
    And our query can, in turn, be split into two components:
    the name of a feature (min-width) and a corresponding value (1024px).
Media Query - Example @media screen and (min-width:1024px) { body: { font-size:100%; } }

Think of a media query like a test for the browser. When a browser reads the stylesheet, the screen and (min-width: 1024px) query asks two questions: first, if it belongs to the screen media type; and if it does, if the browser's viewport is at least 1024 pixels wide. If the browser meets both of those criteria, then the styles enclosed within the query are rendered; if not, the browser disregards the styles.

Media queries can also be placed on link elements by inserting them into the media attribute:

Media Query - Using Link Tag <link rel="stylesheet" href="wide.css" media="screen and (min-width: 1024px)" />

Or we can attach them to @import statements:

Media Query - Using Import Statement @import url("wide.css") screen and (min-width: 1024px)" />

Display Area Vs. Rendering Area

In the spec's language, every device has a 'display area' and 'rendering surface.'

  • Browser's viewport is the display area;
  • Entire display is the rendering surface.

So on a desktop, the display area would be the browser window; the rendering surface would be the desktop screen.

Features that can be queried:

FEATURE NAME DEFINITION HAS min- AND max- PREFIXES
width The width of the display area. yes
height The height of the display area. yes
device-width The width of the device's rendering surface. yes
device-height The height of the device's rendering surface. yes
orientation Accepts portrait or landscape values. no
aspect-ratio Ratio of the display area's width over its height. For example: on a desktop, you'd be able to query if the browser window is at a 16:9 aspect ratio. yes
device-aspect-ratio Ratio of the device's rendering surface width over its height. For example: on a desktop, you'd be able to query if the screen is at a 16:9 aspect ratio. yes
color The number of bits per color component of the device. For example, an 8-bit color device would successfully pass a query of (color: 8). Non-color devices should return a value of 0. yes
color-index The number of entries in the color lookup table of the output device. For example, @media screen and (min-color-index: 256). yes
monochrome Similar to color, the monochrome feature lets us test the number of bits per pixel in a monochrome device. yes
resolution Tests the density of the pixels in the device, such as screen and (resolution: 72dpi) or screen and (max-resolution: 300dpi). yes
scan For tv-based browsing, measures whether the scanning process is either progressive or scan. no
grid Tests whether the device is a grid-based display, like feature phones with one fixed-width font. Can be expressed simply as (grid). no
