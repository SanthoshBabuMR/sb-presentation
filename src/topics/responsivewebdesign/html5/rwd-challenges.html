Optimized HTML Files

Responsive web design is about customizing the html markup, so that the same markup is delivered across devices that access it.

Markup and Web Page Components working perfectly on a desktop version of a website may not be required for mobile version, which could be a video, an embedded map or an image. CSS media queries work incredibly well at hiding things, but not that well at removing them. CSS media queries wont remove any unnecessary script or html that wont be used on mobile or desktop versions. It just leaves it there: hidden.

CSS media queries will not remove unnecessary html nor remove javascript that will slow down mobile browsers. We are now left with increased html file size for small screen device. Increase in files size impacts load time.

FOUT

FOUT known as flash of unstyled text is what we get while using @font-face in Firefox and Opera. As web fonts needs to be downloaded before they are used, browsers takes their own approach.

Before the web fonts are downloaded, firefox and opera renders text in default font; once web fonts are downloaded, firefox and opera renders text using the web font, resulting in FOUT.

Webkit, renders text on screen, only if web fonts are downloaded; before which, it simply hides the text avoiding FOUT bug, but users will have to wait to read any content, until the fonts are downloaded.

IE on the other hand, starts downloading the font file as soon as it encounters the @font-face declaration; however it doesn't render the page, until the web fonts are downloaded if there is a SCRIPT tag above the @font-face declaration.

FOUT can be overcome by using WebFont Loader - a javascript library aiming to provide a number of event hooks giving you a lot of control over the loading.
https://developers.google.com/webfonts/docs/webfont_loader

HTTP request

Mobile versions compared with desktop counterparts, would require lesser number of CSS, JavaScript, Video, Image etc; which implies lesser number of HTTP request.

With responsive design, we link all the resources (css, js, images) in one html file. As a result, mobile versions generate more number of HTTP requests, than required; Similar is the case with desktop devices, which ends up downloading resources required for both large screen and small screen displays.

A simple solution would be to use asynchronous resource loaders such as yepnope, labjs, headjs etc;

Yepnope is a simple resoure loader, that can be used in combination with modernizr to load the appropriate set of CSS and JS depending on screen width and could also be helpful in loading fallback resources only for the browser required.

Image optimization

Image optimization focuses on

  • Loading appropriate images for the corresponding devices.
  • Image resizing

ref:

  • http://24ways.org/2011/adaptive-images-for-responsive-designs-again

With fluid images, we deliver images at the maximum size they will be used at. We dont declare the height and width in html, but instead let the browser resize the images as needed while using CSS to guide their relative size.

Using CSS media queries isnt very bad to handle images and one might think hiding background images could be a viable solution. Well, the display:none wont stop a background image to be downloaded on a mobile device. Other part to consider is that if you use CSS media query to replace the background image with a mobile version; in some cases it would actually download both images. Nonetheless, there are two cases that seem to work: setting the parent of the element with a background image to display:none, and using a min-width declaration to specify a minimum browser width for the desktop image and a max-width for the mobile image. The result of both is only the mobile image been downloaded and viewed on the mobile device.

Image resizing is expensive process. Viewing a responsive website on a mobile device forces the CPU and memory of it while resizing images because you are downloading a large image, uncompressing it in your phones memory and then resizing it to fit a small screen.

Few javascript based techniques have been used to make images responsive, but they seem to be limited in one way or the other

Current/Future approaches to deal with responsive images:

Filament Group Responsive Images:

Delivers optimized, contextual image sizes in responsive layouts that utilize dramatically different image sizes at different resolutions. Ideally, this could enable developers to start with mobile-optimized images in their HTML and specify a larger size to be used for users with larger screen resolutions -- without requesting both image sizes, and without UA sniffing.
http://filamentgroup.com/lab/responsive_images_experimenting_with_context_aware_image_sizing/

DoubleTake

Doubletake dynamically updates the src of images based on the browser width. Start with a small, mobile-friendly image in your HTML. Doubletake will use a defined set of breakpoints to update image SRCs when necessary.
http://www.grahambird.co.uk/lab/doubletake/

Adaptive Images:

Adaptive Images detects the visitor's screen size and automatically creates, caches, and delivers device appropriate re-scaled versions of your web page's embeded HTML images.
http://adaptive-images.com/

Webp:

WebP, developed by Google, is a new image format that provides lossless and lossy compression for images on the web. WebP lossless images are 28% smaller in size compared to PNGs. WebP lossy images are 25-34% smaller in size compared to JPEG images at equivalent SSIM index. WebP supports lossless transparency (also known as alpha channel) with just 22% additionalbytes. WebP also supports features such as animation, ICC color profile, XMP meta-data and tiling.
https://developers.google.com/speed/webp/

Fallback for Older browsers

Fallback for older browsers, includes support for older mobile devices and ie6-8, which do not support CSS3 Media Queries.

Progressive enhancement solves this problem, by establishing a basic level of user experience that all browsers will be able to provide when rendering your web site, but you also build in more advanced functionality that will automatically be available to browsers that can use it.

respond.js plugin can be used achieve media-query effect in IE6-8;
http://code.google.com/p/css3-mediaqueries-js/

However, a better approach would be to serve IE6-8 with a fixed layout CSS. The major trade-off to consider would be the performance. Already several hacks and plugins are being to achieve cross-browser consistency in ie6-8. Trying to emulate media queries could impact the performance and make the coding process complicated, which adds up to development time and cost.
