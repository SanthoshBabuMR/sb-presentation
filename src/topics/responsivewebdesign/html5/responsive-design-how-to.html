Graceful degradation Vs. Progressive Enhancement

Graceful degradation is the practice of building web functionality so that it provides a certain level of user experience in more modern browsers, but it will also degrade gracefully to a lower level of user in experience in older browsers. This lower level is not as nice to use for your site visitors, but it does still provide them with the basic functionality that they came to your site to use; things do not break for them.
Progressive enhancement is similar, but it does things the other way round. You start by establishing a basic level of user experience that all browsers will be able to provide when rendering your web site, but you also build in more advanced functionality that will automatically be available to browsers that can use it.

In other words, Graceful degradation is the journey from complexity to simplicity, whereas Progressive Enhancement is the journey from simplicity to complexity. Progressive enhancement is considered a better methodology than Graceful degradation because it tends to cover a greater range of potential issues as a baseline.

With responsive web design, we cover a wide range of devices, so we need to start out with a solid and simple design structure. Moreover, With CSS3 features available in most modern browsers, we are now capable of having rounded-corners, gradients, multiple backgrounds, text-shadow, box-shadow using CSS.

A better approach to dealing CSS3 is that, we could start out with a basic layout for CSS3 non-supporting browsers providing them with minimal UI Styling. As we move towards targeting CSS3 enabled browsers, we could have all kinds of CSS3 features in action.

Identify Break Points

Identifying the breakpoints, helps us to know the number of layouts we will be working with.

As they are an ever-increasing number of devices with different screen resolutions to take into account with a responsive design, we identify the breakpoints by classifying the devices based on their width.

This ensures our pages are independent of the type of device that accesses it.

Create wireframes

Once we are done with breakpoints, we can now create basic wireframes of how the design will look at the respective break points. We don't intend to create a rock solid wireframe design, but rather a simple wireframe structure that focuses primarily on layout and component position in the web page.

Following are some of the important design decisions this phase focuses on,

  • Type of layout at each break point i.e. 2 column/3 column/single column layout
  • How to present the Global Navigation
  • How do we handle interactivity added using JS across different layouts?
    1. Although a responsive site will only change the CSS depending on the width; if there are complicated elements that rely heavily on JavaScript, they may not translate well on a smaller device and it could be worth hiding these.
  • Designing form layouts, webpage components for different screen widths.
  • As only one markup is served across different devices, there should be no duplicate content or components.

Once again, the key to a responsive design is progressive enhancement, where we start designing for a low-end devices and enhance as we move towards to high-end devices.

Framework

Responsive Framework helps to quickly design prototype and get it working.

These frameworks divide the layout into a system of grids consisting of columns & gutters. We will have to set corresponding class names as required by the framework, to get our system of grids working.

Choosing a framework makes our layout process simplified; if we find that, frameworks are not suiting our requirements, then we can build our own framework.

Viewport for iOS

Mobile Safari presents desktop-sized sites on small screens by rendering to a virtual browser screen that is 980px wide, and letting you pan and zoom that page. Viewport tags let you tell Mobile Safari that your site displays properly narrower than 980px, and set other preferences for scaling your page.

Setting viewport width to the width of the device <meta name="viewport" content="width=device-width"> Setting the initial scale to 1.0 <meta name="viewport" content="initial-scale=1.0"> Setting the initial scale and to turn off user scaling <meta name="viewport" content="initial-scale=2.3, user-scalable=no"> Putting it together, Setting width to device-width and disable user scaling, <meta name="viewport" content="width=device-width, initial-scale=1, maximum-scale=1">

Typography - EM Vs. pixel

Pixels are fixed-size units that are used in screen media. One pixel is equal to one dot on the computer screen (the smallest division of your screen's resolution). With sizing fonts in pixels, we are literally telling browsers to render the letters exactly that number of pixels in height. Pixels don't work well with accessibility and have usability concerns. In IE 6 and below, font-sizes set in pixels cannot be resized by the user.

em refers to font size of the element; ems allows designs to be resolution independent. 1em is equal to the current font-size of the element. If we haven't set font size anywhere on the page, then it would be the browser default, which is probably 16px. So by default 1em = 16px. If we were to set a font-size of 20px on your body, then 1em = 20px.

Using EM:
When working with EM, we set the base-font to the body tag in px / em / %.

body { font-size: 16px; } /* default font-size of browser */
body { font-size:100%; } /* 16px/16px; font-size of browser default's to 16px */

now, if we want our elements, say div to be at 18px, then we target it as,

div { font-size: 0.875em } /* 18px/16px = 1.125em*/

As with fluid layout, the simple math calculation is used here,

target / context = result

target = font-size required for the element;
contenxt = font-size of the containing parent element;

However, em-based font sizing leads to compounding effect.

With the above code, if we had child div inside another parent div,

The effective font-size of child div would be, [1.125em * 1.125em], which would be 20px; if we had another div within the child div, then the font-size of this div would be [1.125em * 1.125em * 1.125em], which would be 23px;

To avoid the compounding effect, we need to be specific in targeting the elements and use target context = result math as required.

CSS3 @font-face

@font-face is a CSS rule which allows to show a font on a Web page even if that font is not installed on the users' computer. Using this, we can get around the problem of web-safe fonts as well as prevent creating additional dependencies in other methods such as Cufón, sIFR, etc.

@font-face @font-face { font-family: <a-remote-font-name>; src: <source> [,<source>]*; [font-weight: <weight>]; [font-style: <style>]; } Example - @font-face usage @font-face { font-family: 'MyWebFont'; src: url('MyWebFont.eot'); /* IE9 Compat Modes */ src: url('MyWebFont.eot?') format('embedded-opentype'), /* IE6-IE8 */ url('MyWebFont.woff') format('woff'), /* Modern Browsers */ url('MyWebFont.ttf') format('truetype'), /* Safari, Android, iOS */ url('MyWebFont.svg#svgFontName') format('svg'); /* Legacy iOS */ }

Web Font Formats

- EOT, TTF, WOFF, SVG

EOT [Embedded OpenType]:

Embedded OpenType is a proprietary standard supported exclusively by Internet Explorer.
The font files are made small in size by use of subsetting (only including the needed characters), and by data compression.

TTF [TrueType Font]:

TrueType is an outline font standard originally developed by Apple Computer. TrueType has become the most common format for fonts on both the Mac OS and Microsoft Windows operating systems.

The primary strength of TrueType was originally that it offered font developers a high degree of control over precisely how their fonts are displayed, right down to particular pixels, at various font sizes.

WOFF [Web Open Font Format]:

The WOFF is a font format for use in web pages. WOFF is in the process of being standardized as a recommendation by the W3C which published WOFF as a working draft back in July 2010.
WOFF is a wrapper that contains TrueType and OpenType fonts, and its not really a new format of its own.

SVG [Scalable Vector Graphics]:

SVG fonts are text files that contain the glyph outlines represented as standard SVG elements and attributes, as if they were single vector objects in the SVG image.
SVG is the only format that can be used for the iPhone and iPad prior to iOS 4.2.

Video

Video(s) can be made responsive similar to images using CSS max-width; But, this applies only to video embedded using html5 video/audio tag, but it doesnt work with embed code using iframe or object tag

So, we have a CSS solution and jquery solution to get around this.

The css solution requires a wrapper container around the iframe containing the video; we now adjust the iframe as well as wrapper container using CSS to make it elastic.
http://www.alistapart.com/articles/creating-intrinsic-ratios-for-video/

CSS solution doesnt address the aspect ratio issue with video; as an alternative we could use javascript.

jquery plugin aims at capturing the aspect ratio of the iframes containing the video on page load, and saves it. Now when the browser is resized, all the videos are resized to fill the available width and maintain their aspect ratio. The javascript solution is limited to youtube, vimeo videos as they are contained within iframes;
http://daverupert.com/2011/09/responsive-video-embeds-with-fitvids/

Media query listeners

A javascript API for responding to media query changes.

Media query listeners help us to evaluate a media query at run time using JavaScript, and subscribing listeners to changes in the media query's evaluation. Support for Media query listeners is very limited.

Instead of having to poll for changes or load several versions of a resource, we can use the API, for example, to download images only of a particular size when a media query match is triggered.

It involves creating a MediaQueryList object using MatchMedia function of Window Object. - The MatchMedia function takes one or more queries as arguments.

Example: Media query listeners mql = window.matchMedia("(max-width: 320px)") mql -> MediaQueryList Object

The result of the above function would be the MediaQueryList Object;

MediaQueryList object has 2 properties matches and media and 2 methods addListener() and removeListener()

mql.matches - returns "true" if the list of media queries matches the state of the current window;

mql.media - returns the serialized version of the media query list, which was used to create the MediaQueryList object.

mql.addListener() - this function is called whenever the evaluation of the media query changes.

mql.removeListener() - removes the given callback function for the corresponding mql object
